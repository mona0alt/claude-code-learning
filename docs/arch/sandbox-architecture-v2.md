# 🔒 Claude Code 沙箱是怎么实现的？

**一文读懂 Linux 沙箱技术核心原理**

---

作为一个 AI 编程助手，Claude Code 需要在用户机器上执行命令。但你有没有想过：

> **它怎么确保 Claude 只能操作你的项目文件，而不会乱动系统其他部分？**

今天我们就来深入剖析 Claude Code 的沙箱实现。内容有点硬核，建议收藏。

## 2. 沙箱的生命周期

沙箱管理分为三个阶段：

### 阶段一：初始化（启动时）

```
REPL 启动
    ↓
SandboxManager.initialize()
    ↓
┌─ 检查是否启用沙箱
├─ 解析 Git worktree 路径
├─ 转换配置参数
└─ 启动网络代理 + 订阅设置变更
```

### 阶段二：命令执行

```typescript
// Shell.ts 中的关键代码
if (shouldUseSandbox) {
  commandString = await SandboxManager.wrapWithSandbox(
    commandString,
    sandboxBinShell,
  )
}
```

### 阶段三：清理（命令完成后）

```
┌─ BaseSandboxManager.cleanupAfterCommand()
│   └─ 清理 bubblewrap 遗留文件
│
└─ scrubBareGitRepoFiles()
    └─ 删除可能被植入的裸 git 文件
```

## 3. 什么是 bubblewrap？

bubblewrap 是 Linux 上最重要的无特权沙箱工具。它不需要 root 权限，就能创建类似容器的隔离环境。

### 核心原理：Linux Namespace

namespace 就像给进程戴上了一副"滤镜"，让它只能看到特定范围的资源。

| Namespace | 作用 | 对应参数 |
|-----------|------|---------|
| PID | 隔离进程 ID | `--unshare-pid` |
| Network | 隔离网络 | `--unshare-net` |
| Mount | 隔离文件系统挂载点 | `--bind`, `--ro-bind` |
| UTS | 隔离主机名 | `--unshare-uts` |
| IPC | 隔离共享内存/信号量 | `--unshare-ipc` |
| User | 隔离用户/组 ID | `--userns` |

### 一个典型的 bubblewrap 命令

```bash
bwrap \
  --unshare-user \
  --unshare-pid \
  --unshare-net \
  --ro-bind /usr /usr \
  --bind /home/user/project /project \
  --tmpfs /tmp \
  --proc /proc \
  --dev /dev \
  firefox
```

这段命令的意思是：

- 创建独立的用户、PID、网络命名空间
- /usr 目录只读挂载
- /home/user/project 映射到 /project（可读写）
- /tmp 使用内存文件系统（进程结束后消失）
- firefox 只能在这些限制内活动

## 4. bubblewrap 的文件系统视图

这是理解沙箱的关键：**沙箱内看到的文件系统，不是完全独立的，而是宿主机的"剪辑版"**。

```
┌─────────────────────────────────────────────────────────────────┐
│                        宿主机文件系统                             │
│                                                                  │
│   /home/user/project    /tmp    /usr    /etc    /root         │
└─────────────────────────────────────────────────────────────────┘
                              │
                              │ 绑定挂载 (bind mount)
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│                       bubblewrap 沙箱内                           │
│                                                                  │
│   /project (可见)    /tmp     /usr    [不可见]  [不可见]       │
│   (可读写)          (tmpfs)  (只读)                            │
└─────────────────────────────────────────────────────────────────┘
```

### Claude Code 的文件系统配置

```typescript
// 允许写入：工作目录 + 临时目录
allowWrite: ['.', '/tmp/claude-xxx']

// 拒绝写入：保护系统文件
denyWrite: [
  '~/.claude/settings.json',    // 设置文件
  '.claude/skills',             // skills 目录
  'HEAD', 'objects', 'refs',    // 裸 git 文件
]
```

**效果**：

```
$ echo "malicious" > /etc/passwd
Permission denied ✓

$ echo "正常" > ./myproject/file.txt
成功 ✓
```

## 5. seccomp：系统调用过滤器

光有 namespace 不够，还需要 **seccomp** 来限制进程能调用的内核接口。

### 两种模式

| 模式 | 描述 |
|------|------|
| 严格模式 | 只允许 read/write/exit/sigreturn，几乎没人用 |
| **seccomp-bpf** | 自定义规则过滤，主流方案 |

### 工作原理

```
┌──────────────────────────────────────────────┐
│           进程调用 mount()                   │
└─────────────────┬────────────────────────────┘
                  ↓
┌──────────────────────────────────────────────┐
│           seccomp 过滤器检查                   │
│    ┌───────────────────────────────────┐    │
│    │ if (syscall in whitelist):        │    │
│    │     allow                         │    │
│    │ else:                             │    │
│    │     kill process (SIGKILL)        │    │
│    └───────────────────────────────────┘    │
└──────────────────────────────────────────────┘
```

### bubblewrap 默认阻止的系统调用

```
文件操作: mount, umount, umount2
进程管理: ptrace, capget, capset
命名空间: setns, unshare, clone (某些flag)
内存管理: mbind, migrat_pages
I/O 操作: ioprio_get, ioprio_set
```

## 6. 网络隔离：域名白名单如何实现？

Claude Code 不仅限制文件访问，还限制网络请求。

### 核心机制：沙箱代理

```
┌─────────────────────────────────────────────────────────────────┐
│                         沙箱内                                   │
│                                                                  │
│      curl https://evil.com                                      │
│              ↓                                                  │
│      连接 127.0.0.1:HTTP_PROXY_PORT                           │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│                       沙箱网络代理                               │
│                                                                  │
│      ┌─────────────────────────────────────────────────┐        │
│      │           检查 Host 头中的域名                   │        │
│      │                                                 │        │
│      │         allowedDomains 匹配？                   │        │
│      │                                                 │        │
│      │    Yes ───→ 转发到真实目标                      │        │
│      │    No  ───→ 返回 HTTP 403                      │        │
│      └─────────────────────────────────────────────────┘        │
└─────────────────────────────────────────────────────────────────┘
```

### 配置示例

```typescript
sandbox: {
  network: {
    allowedDomains: [
      'api.anthropic.com',
      '*.github.com'
    ],
    deniedDomains: ['evil.com']
  }
}
```

### 多层防护

```
┌─────────────────────────────────────────────────────────────────┐
│ 1. network namespace    → 物理上无网络接口                       │
├─────────────────────────────────────────────────────────────────┤
│ 2. HTTP/SOCKS 代理     → 域名过滤                                │
├─────────────────────────────────────────────────────────────────┤
│ 3. 环境变量注入         → 命令自动走代理                          │
├─────────────────────────────────────────────────────────────────┤
│ 4. seccomp             → 阻止 raw socket                        │
├─────────────────────────────────────────────────────────────────┤
│ 5. SSRF Guard          → 阻止私有 IP 访问                       │
└─────────────────────────────────────────────────────────────────┘
```

## 7. 三层防御体系总结

```
┌─────────────────────────────────────────────────────────────────┐
│                      文件系统隔离                                │
│                                                                 │
│   ✓ allowWrite: 仅工作目录和临时目录                            │
│   ✓ denyWrite: 保护 settings.json、skills、git 文件             │
│   ✓ denyRead: 敏感路径                                         │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼ AND
                              │
┌─────────────────────────────────────────────────────────────────┐
│                       网络隔离                                   │
│                                                                 │
│   ✓ allowedDomains: 域名白名单                                   │
│   ✓ network namespace: 无网络接口                                │
│   ✓ 沙箱代理: 拦截并过滤请求                                     │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼ AND
                              │
┌─────────────────────────────────────────────────────────────────┐
│                      系统调用过滤                                │
│                                                                 │
│   ✓ seccomp: 阻止 mount/ptrace/setns 等危险调用                │
│   ✓ capabilities: 丢弃特权                                       │
└─────────────────────────────────────────────────────────────────┘
```

## 8. 最后

Claude Code 的沙箱设计非常精妙：

```
          ┌─────────────────────────────────────────┐
          │  bubblewrap: namespace 级别资源隔离     │
          │      PID / Network / Mount / UTS        │
          └─────────────────┬───────────────────────┘
                            │
          ┌─────────────────┴───────────────────────┐
          │  seccomp: 系统调用级别安全过滤          │
          │      阻止危险内核接口                    │
          └─────────────────┬───────────────────────┘
                            │
          ┌─────────────────┴───────────────────────┐
          │  网络代理: 域名级别访问控制             │
          │      allowedDomains 白名单              │
          └─────────────────────────────────────────┘
```

三层防护相互配合，既让 Claude 能正常工作，又有效防止了越界操作。
